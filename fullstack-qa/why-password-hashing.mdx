## 🛑 El Error del Junior

Guarda la contraseña directamente en la base de datos como string — cuando hay una brecha de seguridad, todas las credenciales quedan expuestas inmediatamente.

## 🧠 El Enfoque Curse Dev (Contra-pregunta)

* **Tú:** *"Storing plain text passwords is one of the most critical security mistakes a developer can make. Would it help if I explained why hashing prevents the damage even if the database is compromised?"*

## 💻 La Respuesta Técnica

**El problema con texto plano:**

```python
# ❌ NUNCA hacer esto
@app.post("/register")
async def register(user_data: UserCreate, db: Session = Depends(get_db)):
    new_user = User(
        email=user_data.email,
        password=user_data.password  # ← Texto plano en la DB
        # Si la DB se compromete → todas las contraseñas expuestas
        # Si el mismo usuario usa la misma contraseña en Gmail → esa también
    )
    db.add(new_user)
    db.commit()
```

**El resultado de una brecha con texto plano:**
```
DB comprometida → atacante obtiene:
- usuario@gmail.com : MiPassword123
- otro@hotmail.com  : password456
→ Puede hacer "credential stuffing" en Gmail, Instagram, bancos
```

**La solución — hashing con bcrypt:**

```python
from passlib.context import CryptContext

# bcrypt es el estándar — slow by design para dificultar brute force
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def hash_password(plain_password: str) -> str:
    return pwd_context.hash(plain_password)
    # "MiPassword123" → "$2b$12$randomsalt...hashedvalue" (irreversible)

def verify_password(plain_password: str, hashed_password: str) -> bool:
    return pwd_context.verify(plain_password, hashed_password)
    # Compara sin "deshacer" el hash

@app.post("/register", status_code=201)
async def register(user_data: UserCreate, db: Session = Depends(get_db)):
    new_user = User(
        email=user_data.email,
        hashed_password=hash_password(user_data.password)  # ✅
    )
    db.add(new_user)
    db.commit()

@app.post("/login")
async def login(credentials: LoginRequest, db: Session = Depends(get_db)):
    user = db.query(User).filter(User.email == credentials.email).first()
    if not user or not verify_password(credentials.password, user.hashed_password):
        raise HTTPException(status_code=401, detail="Invalid credentials")
    return {"access_token": create_jwt(user.id)}
```

**Por qué bcrypt es seguro:**
- Es **one-way**: el hash no se puede revertir a la contraseña original
- Incluye **salt** automático: mismo password = hashes distintos cada vez
- Es **lento intencionalmente**: hace costoso el brute force masivo

## 🇺🇸 El Script en Inglés (Para la entrevista)

> *"Storing passwords in plain text is a critical security failure because if the database is ever compromised—through a SQL injection, a backup leak, or an insider threat—every single user's password is immediately exposed. The problem is compounded by password reuse: many users use the same password across multiple services, so one database breach can cascade into Gmail, banking, or other accounts. The solution is one-way hashing with an algorithm like bcrypt, which is specifically designed to be slow, making brute-force attacks computationally expensive. Bcrypt also automatically applies a unique salt to each hash, so even two users with identical passwords produce different hash values. During login, instead of comparing passwords directly, you compare the plain text against the stored hash using a verify function—so the original password never needs to be recovered or stored."*

## 🔑 Vocabulario Clave

* **Password hashing:** Transformación irreversible de una contraseña en un hash seguro
* **bcrypt:** Algoritmo de hashing diseñado para ser lento y seguro — estándar para contraseñas
* **Salt:** Valor aleatorio añadido antes de hashear para evitar hashes idénticos
* **Credential stuffing:** Ataque que usa credenciales filtradas en otros servicios
* **One-way function:** Función matemática que no puede invertirse para recuperar el input
