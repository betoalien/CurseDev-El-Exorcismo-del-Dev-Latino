## 🛑 El Error del Junior

Prueba cada capa por separado (el formulario funciona, el endpoint responde, la DB guarda) pero no verifica el flujo completo — el bug está en la integración entre capas y nunca lo detecta.

## 🧠 El Enfoque Curse Dev (Contra-pregunta)

* **Tú:** *"End-to-end flow verification is where integration bugs surface that unit tests miss. Would it help if I traced a complete flow from user action to database and explained what to verify at each layer?"*

## 💻 La Respuesta Técnica

Un flujo **end-to-end (E2E)** funciona cuando una acción del usuario recorre correctamente **todas las capas** de la aplicación sin romperse en ninguna.

**Trazar un flujo completo — Crear una orden:**

```
1. USUARIO INTERACTÚA
   └── Hace clic en "Comprar" en el formulario del frontend

2. FRONTEND VALIDA Y ENVÍA
   └── Valida campos con Zod
   └── fetch("POST /api/orders", { body: JSON, headers: {Authorization} })

3. BACKEND RECIBE Y VALIDA
   └── FastAPI verifica el JWT → identifica al usuario
   └── Pydantic valida el body → rechaza datos inválidos con 422
   └── Llama al servicio de órdenes

4. LÓGICA DE NEGOCIO
   └── Verifica stock disponible
   └── Calcula total con impuestos
   └── Valida crédito del cliente

5. BASE DE DATOS
   └── INSERT en tabla orders
   └── UPDATE stock en tabla products
   └── COMMIT de la transacción

6. SERVICIOS AUXILIARES (background)
   └── Redis: invalida caché de productos
   └── Email: envía confirmación

7. RESPUESTA AL FRONTEND
   └── 201 Created + JSON con la orden creada

8. UI SE ACTUALIZA
   └── Redirige a /order/confirmation
   └── Muestra resumen del pedido
```

**Verificar que el flujo E2E funciona:**

```python
# Test E2E con TestClient de FastAPI (sin mocks)
from fastapi.testclient import TestClient

def test_complete_order_flow(client: TestClient, test_db):
    # 1. Registrar usuario
    client.post("/api/auth/register", json={"email": "test@test.com", "password": "pass123"})

    # 2. Login → obtener token
    login_res = client.post("/api/auth/login", json={"email": "test@test.com", "password": "pass123"})
    token = login_res.json()["access_token"]

    # 3. Crear orden (flujo completo)
    order_res = client.post(
        "/api/orders",
        json={"product_id": 1, "quantity": 2},
        headers={"Authorization": f"Bearer {token}"}
    )

    # 4. Verificar respuesta
    assert order_res.status_code == 201
    order = order_res.json()
    assert order["status"] == "pending"
    assert order["total"] > 0

    # 5. Verificar persistencia en DB
    db_order = test_db.query(Order).filter(Order.id == order["id"]).first()
    assert db_order is not None
    assert db_order.user_id is not None
```

**Síntomas de que el flujo E2E está roto:**
- El frontend muestra éxito pero la DB no tiene el registro
- El backend devuelve 200 pero el frontend muestra error
- Funciona localmente pero falla en producción (variables de entorno)
- La primera vez funciona, las siguientes no (estado no reseteado)

## 🇺🇸 El Script en Inglés (Para la entrevista)

> *"An end-to-end flow working means that a single user action successfully traverses every layer of the application—from the UI interaction, through the HTTP request, backend validation, business logic, database write, and back to the UI with the correct result. It's the integration point where bugs that unit tests miss tend to surface: the endpoint works in isolation, the database query works, the frontend form works, but the combination fails because of a mismatched field name, a missing authorization header, a database transaction not being committed, or an environment variable not set in production. I verify end-to-end flows by tracing the complete path manually first—checking network requests in browser DevTools, backend logs, and database records—and automating it with integration tests that use a real database and don't mock the HTTP layer."*

## 🔑 Vocabulario Clave

* **End-to-end flow:** Flujo completo desde acción del usuario hasta persistencia y respuesta
* **Integration bug:** Error que ocurre en la interacción entre capas, no dentro de una sola
* **Happy path:** El flujo exitoso esperado — el primero que debe verificarse E2E
* **Integration test:** Prueba que verifica la interacción entre múltiples componentes reales
* **Layer boundary:** Punto donde una capa pasa datos a la siguiente — donde ocurren los bugs
