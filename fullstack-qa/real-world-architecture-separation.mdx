## 🛑 El Error del Junior

Mezcla todo en un mismo servidor — frontend, lógica de negocio, queries SQL y llamadas a servicios externos en el mismo archivo, creando un sistema que no puede escalarse ni mantenerse.

## 🧠 El Enfoque Curse Dev (Contra-pregunta)

* **Tú:** *"Separation of concerns at the architecture level is what makes a system maintainable and scalable. Would it help if I drew out what a real-world fullstack architecture looks like and explained what each layer's responsibility is?"*

## 💻 La Respuesta Técnica

Una arquitectura fullstack real separa responsabilidades en capas distintas, donde cada una tiene una función clara.

**Arquitectura de una aplicación fullstack moderna:**

```
┌─────────────────────────────────────────────────────────┐
│                      CLIENTE                             │
│  Next.js (React)                                         │
│  • UI components, routing, formularios                   │
│  • Fetch a la API, estados de carga y error              │
│  • Sin lógica de negocio, sin acceso directo a DB        │
└───────────────────────┬─────────────────────────────────┘
                        │ HTTP / REST (JSON)
┌───────────────────────▼─────────────────────────────────┐
│                      API / BACKEND                       │
│  FastAPI                                                 │
│  • Endpoints, validación con Pydantic                    │
│  • Autenticación / autorización                          │
│  • Lógica de negocio (servicios)                         │
│  • Orquesta las capas inferiores                         │
└──────────┬────────────────────────┬─────────────────────┘
           │ SQL / ORM              │ Redis commands
┌──────────▼──────────┐  ┌──────────▼──────────────────┐
│    BASE DE DATOS     │  │   SERVICIOS AUXILIARES       │
│  PostgreSQL          │  │  Redis (caché/sesiones)      │
│  • Persistencia      │  │  Email service               │
│  • Integridad datos  │  │  Storage (S3)                │
│  • Relaciones        │  │  Task queue (Celery)         │
└─────────────────────┘  └─────────────────────────────-┘
```

**Por qué cada separación importa:**

```python

async def dashboard(request: Request):

    html = "<html><body>"

    # Lógica de negocio en el endpoint
    users = db.execute("SELECT * FROM users WHERE active=1")
    revenue = sum(db.execute("SELECT amount FROM orders"))

    # Llamada directa a servicio externo sin manejo de errores

    return HTMLResponse(html + f"Revenue: {revenue}" + "</body></html>")

# ✅ Cada capa con su responsabilidad
# frontend/app/dashboard/page.tsx — solo UI
async function DashboardPage() {
  const stats = await fetchDashboardStats()  // Llama a la API
  return <DashboardView stats={stats} />
}

# backend/routers/dashboard.py — solo HTTP layer
@app.get("/api/dashboard/stats", response_model=DashboardStats)
async def get_stats(service: DashboardService = Depends()):
    return await service.get_stats()

# backend/services/dashboard_service.py — lógica de negocio
class DashboardService:
    async def get_stats(self) -> DashboardStats:
        revenue = await self.order_repo.total_revenue()
        active_users = await self.user_repo.count_active()
        return DashboardStats(revenue=revenue, users=active_users)
```

**Qué habilita la separación:**
- **Escalabilidad**: El frontend puede desplegarse en un CDN, el backend en más instancias
- **Testing**: Cada capa se testea de forma independiente
- **Seguridad**: La DB nunca es accesible desde el frontend directamente
- **Mantenibilidad**: Cambiar la UI no requiere tocar la lógica de negocio

## 🇺🇸 El Script en Inglés (Para la entrevista)

> *"Separating frontend, backend, database, and auxiliary services is an application of the separation of concerns principle at the architecture level, and it has concrete consequences. The frontend is responsible for rendering UI and orchestrating user interactions—it should never contain business logic or have direct database access. The backend handles the API contract, authentication, and business rules—it's the only layer that talks to the database and other services. The database is responsible for data persistence and integrity. Auxiliary services like Redis, email providers, or storage handle specific cross-cutting concerns. This separation means each layer can scale independently—the frontend can be served from a CDN, the backend can run on multiple instances behind a load balancer, and the database can be scaled vertically. It also means each layer can be tested and deployed independently, and a change to the UI doesn't risk breaking business logic."*

## 🔑 Vocabulario Clave

* **Separation of concerns:** Principio de que cada capa debe tener una responsabilidad única y clara
* **CDN (Content Delivery Network):** Red de distribución para servir el frontend estáticamente
* **Load balancer:** Distribuye requests entre múltiples instancias del backend
* **Service layer:** Capa entre los endpoints y la DB que contiene la lógica de negocio
* **Auxiliary service:** Servicio de soporte (Redis, email, storage) que apoya al backend principal
