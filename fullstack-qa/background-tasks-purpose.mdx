## 🛑 El Error del Junior

Ejecuta todo dentro del endpoint — el usuario espera 5 segundos mientras se envía un email de confirmación, se procesa una imagen o se actualiza un sistema externo, causando timeouts y mala experiencia.

## 🧠 El Enfoque Curse Dev (Contra-pregunta)

* **Tú:** *"Background tasks are about decoupling the user response from work that doesn't need to complete before you respond. Would it help if I showed the pattern in FastAPI and explained which operations belong in the background?"*

## 💻 La Respuesta Técnica

Las **tareas en segundo plano** ejecutan trabajo costoso después de que la respuesta ya fue enviada al usuario, evitando que el usuario espere por operaciones que no afectan la respuesta inmediata.

**Sin background tasks — el usuario espera todo:**

```python
# ❌ El usuario espera 5+ segundos mientras se envía el email
@app.post("/orders", status_code=201)
async def create_order(order_data: OrderCreate, db: Session = Depends(get_db)):
    order = save_order(order_data, db)          # 50ms
    send_confirmation_email(order)              # 2000ms ← el usuario espera esto
    update_inventory_system(order.items)        # 1500ms ← y esto
    notify_warehouse_api(order)                 # 800ms ← y esto
    return order  # Responde después de 4.35 segundos
```

**Con background tasks en FastAPI:**

```python
from fastapi import BackgroundTasks

# Las tareas se ejecutan DESPUÉS de que se envía la respuesta
@app.post("/orders", status_code=201)
async def create_order(
    order_data: OrderCreate,
    background_tasks: BackgroundTasks,
    db: Session = Depends(get_db)
):
    # 1. Hacer solo lo necesario para la respuesta
    order = save_order(order_data, db)  # 50ms

    # 2. Programar el resto para después
    background_tasks.add_task(send_confirmation_email, order)
    background_tasks.add_task(update_inventory_system, order.items)
    background_tasks.add_task(notify_warehouse_api, order)

    # 3. Responder inmediatamente
    return order  # Responde en 50ms — las otras tareas siguen en background
```

**Para tareas más pesadas — cola de mensajes con Redis:**

```python
# Para procesar archivos grandes, importaciones masivas, etc.
# FastAPI BackgroundTasks es simple pero sincrónico en el mismo proceso
# Para mayor robustez: Celery + Redis como broker

from celery import Celery

celery_app = Celery("tasks", broker="redis://localhost:6379/0")

@celery_app.task
def process_uploaded_csv(file_path: str, user_id: int):
    """Se ejecuta en un worker separado, no bloquea el servidor."""
    rows = read_csv(file_path)
    for row in rows:
        save_to_db(row, user_id)
    send_completion_email(user_id)

@app.post("/upload/csv")
async def upload_csv(file: UploadFile, current_user = Depends(get_current_user)):
    path = save_file(file)
    process_uploaded_csv.delay(path, current_user["id"])  # ← Encola la tarea
    return {"message": "Processing started, you'll receive an email when done"}
```

**Qué va en background vs. qué no:**

```
✅ Background                    ❌ No background (necesario para respuesta)
─────────────────────            ─────────────────────────────────────────
Envío de emails                  Guardar el pedido en la DB
Notificaciones push              Validar stock disponible
Procesamiento de imágenes        Verificar pago
Sync con sistemas externos       Calcular el total del carrito
Generación de reportes           Autenticar al usuario
```

## 🇺🇸 El Script en Inglés (Para la entrevista)

> *"Background tasks exist to decouple the HTTP response time from operations that don't need to complete before the user gets their answer. If a user creates an order, they need confirmation that the order was saved—but they don't need to wait for a confirmation email to be sent, inventory to sync, or a warehouse notification to go out. Running all of that inside the request handler forces the user to wait for the slowest operation in the chain, which increases response times and risks timeouts. FastAPI has built-in BackgroundTasks for simple cases: you schedule the task before returning the response, and FastAPI runs it after the response is sent. For heavier workloads—like processing uploaded files or running batch jobs—a proper task queue like Celery with Redis as the broker is more appropriate, because it runs in a separate worker process and can retry failures, limit concurrency, and distribute work."*

## 🔑 Vocabulario Clave

* **Background task:** Tarea que se ejecuta después de enviar la respuesta HTTP al cliente
* **Task queue:** Cola de tareas (ej: Celery) para procesar trabajo en workers separados
* **Message broker:** Sistema intermediario (ej: Redis) que almacena tareas pendientes
* **Worker:** Proceso separado que consume y ejecuta tareas de la cola
* **Response time:** Tiempo hasta que el cliente recibe la respuesta — debe ser mínimo
