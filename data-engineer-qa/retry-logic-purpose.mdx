## 🛑 El Error del Junior

"Para que no se caiga el pipeline si hay un error"—correcto pero no distingue entre errores transitorios (donde reintentar tiene sentido) y errores permanentes (donde no).

## 🧠 El Enfoque Curse Dev (Contra-pregunta)

* **Tú:** *"Retries are powerful but need to be implemented carefully—retrying a permanent error is worse than failing fast. Could I explain how I distinguish between the two and what a proper retry strategy looks like?"*

## 💻 La Respuesta Técnica

Los **reintentos** son útiles para recuperarse de **fallos transitorios**: timeouts, caídas momentáneas de la API, errores de red. No sirven para errores permanentes como credenciales inválidas o recursos inexistentes.

**Implementación con backoff exponencial:**

```python
import time
import requests
import logging

logger = logging.getLogger(__name__)

def fetch_with_retry(url: str, headers: dict, max_retries: int = 3) -> dict:
    """Fetch con reintentos y backoff exponencial."""
    retryable_codes = {429, 500, 502, 503, 504}  # Errores transitorios

    for attempt in range(1, max_retries + 1):
        try:
            response = requests.get(url, headers=headers, timeout=10)

            if response.status_code in retryable_codes:
                wait = 2 ** attempt  # 2s, 4s, 8s...
                logger.warning(f"Attempt {attempt} failed ({response.status_code}). Retrying in {wait}s")
                time.sleep(wait)
                continue

            response.raise_for_status()
            return response.json()

        except requests.exceptions.Timeout:
            logger.warning(f"Timeout on attempt {attempt}")
            if attempt == max_retries:
                raise

    raise Exception(f"All {max_retries} attempts failed for {url}")
```

**Errores donde NO reintentar:**
- `401 Unauthorized` → credencial inválida
- `403 Forbidden` → sin permisos
- `404 Not Found` → recurso no existe

## 🇺🇸 El Script en Inglés (Para la entrevista)

> *"Retry logic makes pipelines resilient against transient failures—temporary conditions like network timeouts, API server overload, or brief outages that would resolve themselves if you waited a moment. The key design decision is distinguishing between retryable and non-retryable errors: a 503 Service Unavailable or a timeout is worth retrying with exponential backoff; a 401 Unauthorized or a 404 Not Found means retrying will never help and you should fail immediately with a clear error message. I implement retries with exponential backoff and a maximum attempt cap—so the pipeline waits 2 seconds, then 4, then 8—rather than hammering a struggling API and making things worse. Libraries like tenacity in Python make this pattern easy to apply consistently across a codebase."*

## 🔑 Vocabulario Clave

* **Transient failure:** Fallo transitorio o temporal
* **Exponential backoff:** Espera creciente entre reintentos (2s, 4s, 8s...)
* **Non-retryable error:** Error permanente que no mejora con reintentos
* **Retry cap:** Límite máximo de intentos
* **Resilience:** Resiliencia del pipeline ante fallos externos
