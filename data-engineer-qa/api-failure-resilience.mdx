## 🛑 El Error del Junior

"Usar try/except"—el manejo de errores es necesario pero no suficiente. La resiliencia real de un pipeline requiere múltiples capas de protección.

## 💻 La Respuesta Técnica

Un pipeline dependiente de APIs externas debe diseñarse asumiendo que **la API va a fallar eventualmente**. La resiliencia se construye en capas:

**Stack completo de prácticas de resiliencia:**

```python
import requests
import time
import json
import logging
from datetime import datetime
from pathlib import Path

logger = logging.getLogger(__name__)

def robust_api_pipeline(api_url: str, api_key: str, output_path: str):

    # 1. TIMEOUT: nunca esperar indefinidamente
    headers = {"Authorization": f"Bearer {api_key}"}

    # 2. REINTENTOS con backoff exponencial
    for attempt in range(1, 4):
        try:
            response = requests.get(api_url, headers=headers, timeout=15)

            # 3. RATE LIMIT: respetar el 429
            if response.status_code == 429:
                wait = int(response.headers.get("Retry-After", 60))
                logger.warning(f"Rate limit. Waiting {wait}s")
                time.sleep(wait)
                continue

            response.raise_for_status()
            data = response.json()

            # 4. VALIDACIÓN DE SCHEMA
            if "orders" not in data:
                raise ValueError("Unexpected schema: 'orders' key missing")

            # 5. ALMACENAMIENTO RAW antes de transformar
            raw_file = Path(output_path) / f"raw_{datetime.now().strftime('%Y%m%d_%H%M%S')}.json"
            with open(raw_file, "w") as f:
                json.dump(data, f)
            logger.info(f"Raw saved: {raw_file}")

            # 6. LOGGING de métricas clave
            logger.info(f"Fetched {len(data['orders'])} orders")
            return data

        except requests.exceptions.Timeout:
            wait = 2 ** attempt
            logger.warning(f"Timeout on attempt {attempt}. Retrying in {wait}s")
            time.sleep(wait)

    # 7. ALERTA si todos los reintentos fallaron
    raise Exception(f"All retries exhausted for {api_url}")
```

**Las 7 capas de resiliencia:**
1. Timeouts explícitos
2. Reintentos con backoff exponencial
3. Manejo de rate limits
4. Validación de schema
5. Almacenamiento raw
6. Logging de métricas
7. Alertas en fallo total

## 🇺🇸 El Script en Inglés (Para la entrevista)

> *"Resilience in an API-dependent pipeline is about designing for failure at every layer, because external APIs will eventually be unavailable, slow, or return unexpected data. My approach combines seven practices: explicit timeouts so the pipeline never hangs indefinitely; retry logic with exponential backoff for transient failures; rate limit handling that respects the provider's 429 responses; schema validation to catch unexpected changes before they corrupt the destination; raw response storage for recovery and auditing; structured logging of key metrics at every step; and alerting when all retries are exhausted. Each of these handles a different failure mode, and skipping any one of them leaves a gap. In production, I've found that the combination of these practices reduces unplanned pipeline downtime by an order of magnitude compared to a simple try/except wrapper."*

## 🔑 Vocabulario Clave

* **Resilience:** Resiliencia del pipeline ante fallos externos
* **Exponential backoff:** Espera creciente entre reintentos (2s, 4s, 8s...)
* **Timeout:** Tiempo máximo de espera antes de cancelar la solicitud
* **Defense in depth:** Múltiples capas de protección contra fallos
* **Unplanned downtime:** Tiempo de inactividad no planificado del pipeline
